Faiss is designed for billion-scale datasets. When your index no longer fits in RAM — or when a single machine cannot handle your query throughput — Faiss provides composable tools to scale out.

## Datasets that don't fit in RAM

### On-disk inverted lists

The most common scaling approach is to keep the coarse quantizer and metadata in RAM while serving the actual vector codes from a memory-mapped file on disk. Faiss's `OnDiskInvertedLists` implements this pattern.

The `faiss.contrib.ondisk.merge_ondisk` helper builds such an index from a set of pre-built shard files:

```python
import faiss
import numpy as np
from faiss.contrib.ondisk import merge_ondisk

d = 128
nlist = 4096

# 1. Train a shared quantizer on a representative sample
quantizer = faiss.IndexFlatL2(d)
trained_index = faiss.IndexIVFPQ(quantizer, d, nlist, 16, 8)
sample = np.random.random((500_000, d)).astype('float32')
trained_index.train(sample)

# 2. Build per-shard indexes (e.g., one per data file)
shard_fnames = []
for i, fname in enumerate(["shard_0.faiss", "shard_1.faiss", "shard_2.faiss"]):
    shard = faiss.clone_index(trained_index)
    shard_data = np.random.random((1_000_000, d)).astype('float32')
    shard.add(shard_data)
    faiss.write_index(shard, fname)
    shard_fnames.append(fname)

# 3. Merge all shard inverted lists into a single on-disk file
merged = faiss.clone_index(trained_index)
merge_ondisk(merged, shard_fnames, "all_shards.ivfdata")
faiss.write_index(merged, "merged_index.faiss")

# 4. Load and search — inverted list data is read from disk via mmap
index = faiss.read_index("merged_index.faiss", faiss.IO_FLAG_ONDISK_SAME_DIR)
index.nprobe = 64

xq = np.random.random((1000, d)).astype('float32')
D, I = index.search(xq, 10)
```

Reads are served page-by-page from the OS page cache, so frequently accessed posting lists are kept in RAM automatically.

### Compressed quantization codes with IVFPQ

PQ compression reduces per-vector storage dramatically. For a 128-dimensional float32 vector (512 bytes), an IVFPQ index with `m=16, nbits=8` stores only 16 bytes per vector — a 32× compression ratio.

**Memory calculation:**

```
RAM per vector = m bytes   (with nbits=8)
               = m/2 bytes (with nbits=4)

Overhead per inverted list: ~8 bytes for list pointer + size
```

| Configuration | Vectors | RAM for codes |
|---|---|---|
| `IndexIVFPQ(d=128, m=16, nbits=8)` | 1 billion | 16 GB |
| `IndexIVFPQ(d=128, m=8, nbits=8)` | 1 billion | 8 GB |
| `IndexIVFPQ(d=128, m=16, nbits=4)` | 1 billion | 8 GB |

<Tip>
Use `nbits=4` to halve memory at a modest recall cost. `IndexPQFastScan` with 4-bit codes provides SIMD-accelerated distance computation that can compensate for the reduced precision.
</Tip>

## Distributed index with client/server

For datasets that span multiple machines, Faiss provides a simple RPC-based client/server framework in `faiss.contrib.client_server`. Each server hosts a shard of the index; the client fans out queries and merges results.

```
Machine 0: faiss-server shard_0.faiss --port 12340
Machine 1: faiss-server shard_1.faiss --port 12341
Machine 2: faiss-server shard_2.faiss --port 12342
```

On the client:

```python
from faiss.contrib.client_server import ClientIndex

# Connect to each shard
shards = [
    ClientIndex("machine-0", 12340),
    ClientIndex("machine-1", 12341),
    ClientIndex("machine-2", 12342),
]

# Search across all shards and merge top-k results
import numpy as np
xq = np.random.random((100, 128)).astype('float32')

all_D, all_I = [], []
for shard in shards:
    D, I = shard.search(xq, k=10)
    all_D.append(D)
    all_I.append(I)

# Merge results (keep global top-k)
import faiss
D_merged, I_merged = faiss.merge_knn_results(
    np.stack(all_D, axis=1),
    np.stack(all_I, axis=1)
)
```

<Note>
`faiss.contrib.client_server` uses Python's `pickle` for serialization via `faiss.contrib.rpc`. This is suitable for internal trusted networks. Do not expose the server port publicly without additional authentication.
</Note>

## Sharding with IndexShards and IndexReplicas

For a single-machine multi-GPU or multi-process setup, Faiss provides two composable wrapper indexes:

### IndexShards

`IndexShards` partitions the vector database across sub-indexes. Each query is sent to all shards and results are merged. Use this when the index does not fit in one GPU's VRAM, or to parallelize search across CPU threads.

```python
import faiss
import numpy as np

d = 64
nb = 200000

xb = np.random.random((nb, d)).astype('float32')

# Create two sub-indexes (shards)
index = faiss.IndexShards(d)
shard0 = faiss.IndexFlatL2(d)
shard1 = faiss.IndexFlatL2(d)

shard0.add(xb[:100000])   # first half
shard1.add(xb[100000:])   # second half

index.add_shard(shard0)
index.add_shard(shard1)

xq = np.random.random((100, d)).astype('float32')
D, I = index.search(xq, 4)
print(I.shape)  # (100, 4)
```

<Note>
IDs in `IndexShards` are local to each shard by default. Set `successive_ids=True` in the constructor to assign globally successive IDs across shards.
</Note>

### IndexReplicas

`IndexReplicas` holds multiple identical copies of the same index, distributing queries round-robin. Use this to increase query throughput when a single index fits in memory but you need to serve many concurrent queries.

```python
import faiss
import numpy as np

d = 64
nb = 100000
xb = np.random.random((nb, d)).astype('float32')

# Build and populate the sub-index
sub_index = faiss.IndexFlatL2(d)
sub_index.add(xb)

# Replicate across 4 CPU threads
replica_index = faiss.IndexReplicas()
for _ in range(4):
    replica_index.addIndex(faiss.clone_index(sub_index))

xq = np.random.random((1000, d)).astype('float32')
D, I = replica_index.search(xq, 4)
```

`index_cpu_to_all_gpus` returns an `IndexReplicas` internally, with one copy per GPU.

## Big batch search

When you have a large query set *and* a large IVF index, standard search loads the same posting lists repeatedly for different queries. `faiss.contrib.big_batch_search` amortizes this cost by grouping queries that fall into the same Voronoi cell and processing all of them together.

```python
from faiss.contrib.big_batch_search import big_batch_search
import faiss
import numpy as np

d = 128
nlist = 4096

# Build a large IVF index
quantizer = faiss.IndexFlatL2(d)
index = faiss.IndexIVFFlat(quantizer, d, nlist, faiss.METRIC_L2)
xb = np.random.random((5_000_000, d)).astype('float32')
index.train(xb)
index.add(xb)

# Large query set
xq = np.random.random((500_000, d)).astype('float32')
index.nprobe = 32

# Big-batch search: results written to output arrays
D, I = big_batch_search(index, xq, k=10, verbose=True)
print(D.shape, I.shape)  # (500000, 10) (500000, 10)
```

<Tip>
`big_batch_search` is most beneficial when `nprobe` is large (≥ 32) and the number of queries exceeds 100K. For smaller query sets, ordinary `index.search` is faster.
</Tip>

## Memory vs accuracy tradeoffs

### PQ compression ratio

PQ replaces each float32 vector with `m` byte codes. The compression ratio is:

```
compression ratio = (d × 4 bytes) / (m bytes)
                  = 4d / m
```

For d=128 and m=16: ratio = 32× (512 bytes → 16 bytes per vector).

With `nbits=4`, the ratio doubles to 64×.

**Recall vs m (d=128, IndexIVFPQ, nprobe=64):**

| m | Bytes/vector | Typical R@1 |
|---|---|---|
| 64 | 64 | ~95% |
| 32 | 32 | ~90% |
| 16 | 16 | ~82% |
| 8 | 8 | ~70% |

### Impact of nprobe on recall and speed

`nprobe` controls the recall–speed tradeoff for IVF-based indexes. Each doubling of `nprobe` roughly doubles search time while providing diminishing recall gains.

| nprobe | Relative time | Typical R@1 (IVFPQ, nlist=4096) |
|---|---|---|
| 1 | 1× | ~30% |
| 8 | ~5× | ~60% |
| 32 | ~15× | ~80% |
| 128 | ~50× | ~90% |
| nlist | brute force | ~100% |

<Note>
These figures are illustrative. Actual recall depends on dataset intrinsic dimensionality, index configuration, and nlist. Always measure recall against ground-truth results for your specific dataset.
</Note>

## Performance tips

**Batch sizing:**
- Search in batches of at least 64 queries. Faiss parallelizes over queries using OpenMP; small batches do not saturate all threads.
- For GPU indexes, batches of ≥ 1024 queries are recommended for peak throughput.

**OpenMP threads:**
- Faiss uses OpenMP for parallel CPU search. Set the thread count with:

  ```python
  faiss.omp_set_num_threads(8)
  ```

  or via the `OMP_NUM_THREADS` environment variable before import.

**Parallel search:**
- Multiple threads can call `index.search` concurrently on the same flat or IVF index. Faiss indexes are read-thread-safe after `add` and `train` are complete.
- `IndexShards` runs each shard in a separate thread when `threaded=True` (the default).

**Memory-mapped indexes:**
- Warm up the OS page cache before serving traffic by reading through the `.ivfdata` file once:

  ```python
  import os
  with open("all_shards.ivfdata", "rb") as f:
      while f.read(1 << 20):  # read in 1 MiB chunks
          pass
  ```

**Choosing nlist:**
- Rule of thumb: `nlist ≈ sqrt(n)` for balanced recall and speed.
- For billion-scale datasets: `nlist = 65536` or `nlist = 262144`.
- Always provide at least `39 × nlist` training vectors.
